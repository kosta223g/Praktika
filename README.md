# FastAPI Practice Project + Docker + Alembic + AuthX

Цей проєкт є розширеним FastAPI-додатком, який пройшов етапи розвитку від базового API до повноцінного вебсервісу з реляційною базою даних, системою міграцій, завантаженням файлів, генерацією PDF-звітів та безпечною JWT-авторизацією з рольовою моделлю доступу (RBAC).

---

## 🚀 Основний стек технологій

- **Бекенд:** FastAPI, Uvicorn, Python 3.12
- **База даних:** PostgreSQL
- **ORM:** SQLAlchemy (асинхронний режим)
- **Міграції:** Alembic (асинхронний шаблон)
- **Менеджер пакетів:** uv
- **Контейнеризація:** Docker, Docker Compose
- **Авторизація та безпека:** AuthX (JWT-токени), Passlib + Bcrypt (хешування паролів)
- **Файли та звіти:** python-multipart, fpdf2

---

## 🛠️ Функціонал та архітектура проєкту

1. **База даних та міграції (Alembic):**
   - Структура бази даних складається з таблиць: `users`, `categories`, `products`, `orders`, `order_items`.
   - Керування схемою бази даних повністю перенесено на Alembic.
   - Налаштовано сумісність запуску міграцій як локально, так і в Docker-контейнері.

2. **Service Layer (Сервісна архітектура):**
   - Уся логіка взаємодії з базою даних для сутностей винесена у окремий шар сервісів (наприклад, `ProductService`), що розділяє бізнес-логіку та роутери.

3. **Завантаження файлів та PDF-генерація:**
   - Реалізовано асинхронне завантаження файлів на сервер (`POST /files/upload`) із локальним збереженням у каталог `uploads/`.
   - Додано можливість скачування динамічного PDF-каталогу товарів (`GET /products/report`), згенерованого за допомогою `fpdf2` у преміум-дизайні.

4. **JWT Авторизація та RBAC (Контроль доступу):**
   - Реєстрація (`POST /auth/register`) з обов'язковим хешуванням паролів та перевіркою унікальності email.
   - Вхід (`POST /auth/login`) за стандартом OAuth2, що повертає JWT access-токен.
   - Створення товарів (`POST /products/`) обмежено — доступне лише авторизованим користувачам.
   - Видалення товарів (`DELETE /products/{id}`) захищено рольовою моделлю — операція дозволена тільки користувачам із роллю `admin`.

## ⚙️ Інструкція із запуску (Docker Compose)

Найпростіший спосіб запустити весь проєкт разом із базою даних PostgreSQL — використати Docker Compose.

### 1. Запуск контейнерів
Зберіть та запустіть контейнери у фоновому режимі:
```bash
docker compose up --build -d
```
Це автоматично створить та запустить:
* Контейнер бази даних PostgreSQL (порт `5432`).
* Контейнер вебсервера FastAPI (порт `8000`).

### 2. Застосування міграцій бази даних
Застосуйте міграції для створення всіх таблиць:
```bash
docker compose exec web uv run alembic upgrade head
```

### 3. Створення початкових даних (Seeding)
Щоб ви могли створювати товари, у базі даних повинна існувати категорія з `id: 1` (оскільки товари посилаються на категорію). Створіть її за допомогою SQL-запиту:
```bash
docker compose exec db psql -U postgres -d fastapi_db -c "INSERT INTO categories (id, name) VALUES (1, 'Test Category') ON CONFLICT DO NOTHING;"
```

## 💻 Локальний запуск (без Docker)

Якщо ви бажаєте запустити бекенд локально (переконайтеся, що у вас запущений локальний PostgreSQL):

1. **Встановлення залежностей:**
   ```bash
   uv sync
   ```
2. **Налаштування `.env`:**
   Створіть файл `.env` та вкажіть вашу адресу підключення до БД (вказуючи хост `localhost` або `127.0.0.1` замість `db`):
   ```ini
   DATABASE_URL=postgresql+asyncpg://postgres:postgres@127.0.0.1:5432/fastapi_db
   ```
3. **Запуск міграцій:**
   ```bash
   uv run alembic upgrade head
   ```
4. **Запуск сервера:**
   ```bash
   uv run fastapi dev main.py
   ```

## 📝 Використання та тестування API

Після запуску застосунок доступний за адресами:
- **Інтерактивна документація (Swagger UI):** [http://localhost:8000/docs](http://localhost:8000/docs)
- **Альтернативна документація (Redoc):** [http://localhost:8000/redoc](http://localhost:8000/redoc)

### Тестовий сценарій для Swagger:
1. Зареєструйте звичайного користувача (`"role": "user"`) та адміністратора (`"role": "admin"`) в ендпойнті `POST /auth/register`.
2. Залогіньтеся через `POST /auth/login` та отримайте JWT `access_token`.
3. Натисніть кнопку **Authorize** у правому верхньому кутку Swagger, вставте ваш токен і авторизуйтеся.
4. Спробуйте створити продукт у `POST /products/` (доступно будь-якому авторизованому користувачу).
5. Спробуйте видалити продукт у `DELETE /products/{id}`:
   - Під токеном звичайного користувача ви отримаєте помилку **403 Forbidden**.
   - Під токеном адміністратора запит пройде успішно (**204 No Content**).
